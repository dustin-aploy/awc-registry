# Certification Process

## Goal

Certification is the strongest public listing level in the AWC registry, formerly the ARP registry. It is meant to signal that a project did more than self-assert compatibility: it produced reproducible evidence and passed explicit registry review.

In Phase 1, certification targets self-hosted AWC workers and related open-layer projects only.

## Required inputs

A certification submission should include:

1. the public AWC declaration artifact or stable declaration URI;
2. the declaration digest used by the compliance report;
3. a public `AWCComplianceReport` generated by `awc-devkit/test-suite`;
4. the public `awc-devkit/test-suite` version used to generate that report;
5. a detached signature for the published report; and
6. contact or owner information for follow-up review.

Where relevant for self-hosted workers, the declaration may also include optional `metadata.platform_hints` defined on the spec side. The registry may reference those hints indirectly through the published declaration artifact, but it does not redefine them locally.

## Review flow

### Step 1 — Protocol alignment check

The reviewer confirms that the declaration and report both target a released AWC `protocol_version` from `awc-spec`.

### Step 2 — Evidence shape check

The reviewer validates that the submitted report matches the protocol compliance-report schema. The registry does not accept ad hoc report structures.

The reviewer also confirms that the referenced spec and tests are public so third parties can independently inspect the basis for the claim.

### Step 3 — Compatibility-level consistency check

The reviewer checks that the listing tier is consistent with the evidence. For example, a project should not be listed as `certified` if the report status failed or if the report notes only indicate `self-tested`.

### Step 4 — Signature verification

The reviewer verifies the detached signature over the published report bytes using the declared signing method and public key distribution model.

### Step 5 — Issuance

If the checks pass, the reviewer publishes a registry entry under `certified/` with issuance and validity metadata.

## Revocation and downgrade

A certification may be revoked, suspended, or downgraded when:

- the signed report can no longer be verified;
- the report is found to misrepresent the evaluated artifact;
- the project materially changes without recertification; or
- a newer protocol version invalidates the listing’s continued trust posture.
